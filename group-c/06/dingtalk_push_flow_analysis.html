流程概述

钉钉消息推送采用三级推送策略，确保消息的高可靠性送达。主通道使用 sessionWebhook 实现快速推送， 当 Webhook 过期或发送失败时，自动降级到钉钉 Open API 进行推送。整个推送链路支持文本、图片、文件、音频、视频五种消息类型， 并通过完善的会话存储机制支持主动消息推送场景（如定时任务）。

3. 三级推送策略详解

# 第一优先级：当前请求中的 sessionWebhook
session_webhook = meta.get("session_webhook") or meta.get("sessionWebhook")

# 第二优先级：to_handle 解析的 webhook URL
route = self._route_from_handle(to_handle)
session_webhook = route.get("session_webhook")

# 第三优先级：从存储中加载（主动推送场景）
webhook_key = route.get("webhook_key")
webhook_entry = await self._load_session_webhook_entry(webhook_key)
session_webhook = webhook_entry.get("webhook")

# 降级策略：Open API
if not session_webhook:
    await self._send_via_open_api(text, conversation_id, ...)

核心方法详解

1send() - 统一推送入口

消息推送的主入口方法，支持主动推送和回复场景。自动解析 to_handle 获取推送目标， 优先使用当前请求的 sessionWebhook，失败后自动降级到 Open API。

2_send_via_session_webhook() - Webhook推送

通过钉钉 sessionWebhook 发送文本消息。支持 Markdown 和纯文本两种格式， 超过3500字符自动切换为文本格式。返回 bool 表示成功与否。

3_send_via_open_api() - Open API推送

当 Webhook 不可用时的降级方案。根据会话类型选择不同 API： 单聊使用 batchSend，群聊使用 groupMessages/send。需要 Access Token 认证。

4_get_session_webhook_for_send() - 目标解析

解析推送目标地址，支持多种格式：dingtalk:sw:xxx（存储key）、 dingtalk:webhook:url、直接URL。优先使用当前请求上下文中的 webhook。

5_save_session_webhook() - 会话存储

保存用户的 sessionWebhook 及相关元数据，用于后续主动推送。存储在内存和磁盘双写， 支持服务重启后恢复。包含 conversation_id、conversation_type、sender_staff_id。

6_invalidate_session_webhook() - 失效清理

当 Webhook 推送失败时，清除内存中的 webhook URL，但保留其他元数据。 这样后续推送可以继续使用 Open API 降级，避免重复失败请求。

7_get_access_token() - Token管理

获取钉钉 Access Token，带实例级缓存（TTL 1小时）。使用双重检查锁定模式， 避免并发场景下的重复请求。Token 过期前自动刷新。

8send_content_parts() - 多内容推送

支持同时推送文本和媒体内容。先发送文本消息，再逐个上传并发送媒体文件。 失败时自动触发降级流程，确保消息送达。

过期处理机制

sessionWebhook 有过期时间（sessionWebhookExpiredTime），系统在以下情况下会处理过期：

  • 推送失败时：自动清除 webhook，保留元数据供 Open API 降级使用
  • 当前请求无 webhook：不使用存储中的旧 webhook，避免过期问题
  • 主动推送场景：从存储加载，失败后降级到 Open API

关键技术点

1. 异步并发安全

使用 asyncio.Lock() 保护共享资源（session_webhook_store、token缓存）， 确保并发场景下的数据一致性。Token 使用双重检查锁定模式优化性能。

2. Markdown 规范化

钉钉 Markdown 解析器有特殊限制，需进行规范化处理：

  • 列表项前添加空行，避免与前文合并
  • 代码块去除多余缩进
  • 特殊字符转义处理

总结与建议

✅ 设计优势

  • 三级推送策略保证消息送达率，Webhook 失败自动降级
  • 会话 Webhook 持久化存储，支持主动推送和定时任务场景
  • 异步架构设计，支持高并发消息推送
  • 完善的错误处理和日志记录，便于问题排查
  • Token 缓存机制，减少 API 调用开销

⚠️ 注意事项

  • sessionWebhook 有过期时间，长时间未活跃用户无法推送
  • Open API 推送需要配置 robot_code 和相应权限
  • 单聊推送必须有 sender_staff_id，否则无法发送
  • 媒体文件上传依赖网络稳定性，需处理上传失败场景
  • 消息长度超过 3500 字符会切换为纯文本格式
